## Section 2: Coverage Definition

### 1. Line of Code (LOC) Coverage: ãË 65%

The overall LOC coverage target is set at **ãË 65%**, with a higher internal target of **ãË 90%** on core business logic modules. Core logic includes draft generation (`DraftEditorModule/index.tsx` ã `buildEntriesFromLog`, `handleGenerateDraft`), statistics computation (`StatsPanel/index.tsx` ã `computeStats`, coverage calculation), reflection validation (`ReflectionModule/validation.ts` ã `countWords`, `hasRepetition`, `validatePrompt`, `validateReflection`), declaration persistence and edit tracking (`EntryRow.tsx` ã save flow, diff delta), and the IntegrityMonitor warning engine (`IntegrityMonitor/index.ts` ã all four warning conditions). Lower coverage is acceptable on presentational UI components (e.g., `OriginBadge.tsx`, `WarningBanner.tsx`), React context boilerplate (`WarningsContext.tsx`, `ReflectionContext.tsx`), and configuration files. This targeting strategy ensures testing effort is concentrated where bugs are most likely and have the highest user impact ã the logic that generates, validates, and persists student declarations.

### 2. Branch/Path Coverage: ãË 70% on Core Logic

Branch coverage targets **ãË 70%** across core logic modules, ensuring every conditional path is tested for both true and false outcomes. Concrete examples from the requirements include: the empty vs. populated interaction log branch in draft generation (FR-26 ã `if (logs.length === 0)` produces zero entries vs. iterating and creating entries); the coverage threshold branch in StatsPanel (FR-27 ã `if (coverage < 0.6)` triggers discrepancy highlight vs. normal display); the reflection validation branches (FR-30 ã `if (wordCount < 25)` fails vs. passes, `if (hasRepetition)` fails vs. passes, both prompts empty vs. one valid vs. both valid); the entry edit validation branch (FR-29 ã `if (content === '')` rejects vs. accepts); and the origin badge transition branch (FR-32 ã `if (origin === 'auto-generated')` changes to `auto-generated-modified` on edit vs. already modified entries remaining unchanged). Each of these conditional paths has a corresponding positive and negative test case in the test case table.

### 3. Requirement Coverage: 100%

Every functional requirement (FR-26 through FR-32) and every non-functional requirement (NFR-47, NFR-48) has at least one mapped test case, as verified by the traceability matrix in Section 3. Most functional requirements have two or more test cases, including at least one positive/happy-path test and one negative or edge-case test. FR-26 is covered by UT-01, UT-02, IT-01, IT-02, ST-01, and ST-02. FR-30 is covered by UT-05 through UT-08, IT-04, and ST-03. This ensures no requirement can pass review without demonstrated test evidence, and provides defense-in-depth through unit, integration, and system-level coverage of each requirement.

### 4. Function/Method Coverage: ãË 80%

Function and method coverage targets **ãË 80%** across the codebase. Every exported function in core modules must be invoked by at least one test. This includes all validation functions in `ReflectionModule/validation.ts` (`countWords`, `hasRepetition`, `validatePrompt`, `validateReflection`), all event handlers in `DraftEditorModule` (`handleGenerateDraft`, `handleEditEntry`, `handleDeleteEntry`, `handleRegenerateDraft`), all API route handlers in the server (`POST /declarations`, `PATCH /declarations/:id/entries/:entryId`, `POST /declarations/:id/submit`), and all IntegrityMonitor check functions (entry deleted, scope reduction, coverage low, tool missing). Lower function coverage is acceptable for lifecycle hooks and simple pass-through wrappers (e.g., context provider components, event bus re-exports) that carry minimal logic and low defect risk.

### 5. NFR Coverage: At Least One Test per NFR Category

Each non-functional requirement category has dedicated test coverage. **Explainability (NFR-47)** is tested through two approaches: NFR-01 verifies the presence and accuracy of global help content by checking that the HelpSection modal renders all sections from `guidance.json` and that content descriptions match actual application behavior; NFR-02 verifies per-module explainability by confirming that tooltips and hints loaded via the `useGuidance()` hook are rendered in the DOM for each major feature. **Usability (NFR-48)** is tested through three complementary approaches: NFR-03 uses task completion testing to verify the full workflow is completable via keyboard-only navigation; NFR-04 runs automated jest-axe accessibility audits on every rendered module to catch WCAG 2.1 AA violations (labels, contrast, ARIA roles); NFR-05 applies a manual heuristic evaluation against Nielsen's 10 heuristics to identify qualitative usability issues that automated tools cannot detect.
